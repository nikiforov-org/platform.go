# Микросервисная платформа `platform.go` (v1)

## Архитектурная схема

Система построена на принципе Shared Nothing — каждый узел автономен, но связан общей шиной данных и оркестратором. Входной трафик распределяется через внешний Managed Load Balancer.

## Роли компонентов

**1. Транспортный слой (NATS)**

- Cluster Mode: NATS запущен на всех серверах, образуя единое кольцо.
- Service Discovery: Микросервисы не знают IP друг друга. Они общаются через "темы" (subjects), например api.v1.micro-1.health.
- Балансировка: Используется механизм Queue Groups. Если запущено 3 экземпляра сервиса, NATS распределит нагрузку между ними автоматически.

**2. Слой оркестрации (Nomad)**

- Hybrid Mode: Каждый агент работает и как server (управление), и как client (исполнение).
- Raw Exec: Используется драйвер прямого запуска бинарников. Это экономит до 100 МБ RAM на каждом узле по сравнению с Docker.
- Self-Healing: При падении бинарника или зависании (проверка через /health), Nomad перезапускает его.

**3. Прикладной слой (Go)**

**3.1. API Gateway (Go): HTTP-to-NATS Bridge (Port 8080)**
- Единственная точка входа. Принимает HTTP-запросы напрямую от Managed Load Balancer.
- Выполняет nc.Request() в шину NATS и возвращает ответ клиенту.
- Stateless: не хранит состояние, развернут на каждой ноде.
- При недоступности NATS возвращает 503 статус.
- `/health` — health check вне middleware-цепочки; используется Nomad для Self-Healing.
- Rate Limiting: три уровня — per-IP общий лимит, per-IP лимит для `/v1/xauth/*` (защита от брутфорса), глобальный счётчик WebSocket-соединений.
- Panic Recovery: падение одного NATS-обработчика не роняет весь сервис; ошибка логируется, клиент получает 500.

**3.2. Microservices:** 
Изолированные бинарники. Не слушают внешние порты. Подключаются только к локальному NATS (Port 4222).

Масштабирование микросервисов осуществляется горизонтально — через запуск нескольких инстансов одного и того же сервиса. Балансировка нагрузки между ними выполняется средствами NATS Queue Groups, без участия gateway.

**3.3. Демонстрационные сервисы (x\*)**

В репозитории присутствуют три сервиса с префиксом `x`: `xhttp`, `xauth`, `xws`. Они являются **демонстрационными примерами** и предназначены исключительно для того, чтобы показать структуру, организацию кода и возможности платформы:

- `xhttp` — пример микросервиса с CRUD-операциями через PostgreSQL и кэшированием через NATS KV.
- `xauth` — пример JWT-аутентификации с HttpOnly-куками и хранением refresh-токенов в NATS KV.
- `xws` — пример управления WebSocket-сессиями через NATS Pub/Sub.

**Эти сервисы не рекомендуются для использования в продакшене.** В реальных проектах на базе данной платформы они заменяются собственными микросервисами, реализующими бизнес-логику конкретного приложения.

## Работа с состоянием (State Management)

Вместо внедрения отдельного Redis-сервера, для кэширования и хранения временных данных (сессии, токены, промежуточные состояния) используется встроенный функционал NATS JetStream (Key-Value Store).

Почему NATS KV вместо Redis:
1. Zero RAM Overhead: Не требует дополнительной оперативной памяти, так как использует уже запущенный инстанс NATS.
2. Native Go Integration: Библиотека nats.go позволяет работать с хранилищем в нативном стиле: `kv.Put(key, value)`, `kv.Get(key)`.
3. Replication & HA: Данные автоматически реплицируются между всеми тремя узлами кластера (Raft-консистентность).
4. Watch Mechanism: Микросервисы могут "подписываться" на изменения конкретных ключей в хранилище, что удобно для мгновенного обновления конфигураций.

Конфигурация хранилища:
- Bucket Type: Key-Value.
- Storage: `File` (с ограничением по размеру на диске) для персистентных данных или `Memory` для максимально быстрого кэша.
- History: Хранение последних 1-5 изменений ключа (настраивается для аудита).

## Распределение ресурсов (Hard Limits)

Чтобы избежать срабатывания OOM Killer, решено установить следующие лимиты на каждом узле:

| Компонент | RAM Limit | Описание |
|-|-|-|
| OS & Kernel | 150 MB | Резервируется через nomad.client.reserved |
| Nomad + NATS | 100 MB | Системные демоны |
| Apps (Total) | Остаток | Доступно для Go-бинарников |

## Локальная разработка

Dev-окружение с Nomad запускается одной командой:

```bash
./deployments/mode/dev/start.sh      # 1 нода
./deployments/mode/dev/start.sh 3    # 3 ноды (NATS-кластер + Nomad-кластер)
./deployments/mode/dev/start.sh stop # остановить
```

Подробная инструкция, проверка self-healing и rolling update — в `deployments/mode/dev/dev.md`.

## Схема CI/CD (GitHub Actions)

Деплой происходит без участия Docker Registry, напрямую на серверы.

1. Сборка: GitHub Actions собирает бинарники: `GOOS=linux GOARCH=amd64 go build`.
2. Доставка: Через `scp-action` бинарники копируются на все сервера в `/usr/local/bin/`.
3. Обновление: GitHub Actions через SSH выполняет `nomad job run` на одном из серверов.
4. Rolling Update: Nomad по очереди перезапускает процессы на узлах, обеспечивая Zero Downtime.

## Сетевые доступы (Firewall)

Для корректной работы кластера между серверами должны быть открыты порты:

| Порт | Протокол |	Назначение |
|-|-|-|
| 4222 | TCP | Доступ сервисов к NATS |
| 6222 | TCP | Кластеризация NATS (между узлами) |
| 4646 | TCP | Nomad HTTP API (для GitHub Actions) |
| 4647-4648 | TCP/UDP |	Внутренний трафик Nomad (RPC/Serf) |
| 80 | TCP | Внешний HTTP трафик (API Gateway) |

## Рекомендации по эксплуатации

1. Логирование:
Все сервисы пишут структурированные JSON-логи в `stderr` через `zerolog`. Уровень задаётся переменной окружения `LOG_LEVEL` (значения: `debug`, `info`, `warn`, `error`; по умолчанию `info`). Ротацию логов настраивать на стороне Nomad (`logs { max_files = 5, max_file_size = 10 }`): файлы пишутся автоматически, дополнительных агентов не требуется.

2. Swap: 
Рекомендуется создать swap-файл на каждой VPS как "подушку безопасности" для кратковременных пиков RAM.

3. Разные ОС: 
При сборке в Go использовать `CGO_ENABLED=0`, чтобы бинарники были статическими и не зависели от версий библиотек `libc` в разных дистрибутивах (Ubuntu/CentOS).

4. Балансировка через Managed LB: 
Cloudflare направляет трафик на единый IP балансировщика. Отказоустойчивость обеспечивается активными Health-checks на стороне балансировщика, который исключает нерабочие узлы из ротации в реальном времени.

5. Отказоустойчивость: 
Падение одной ноды считается штатной ситуацией. При падении двух нод NATS-кластера или потере кворума JetStream возможна деградация state-слоя и частичная недоступность системы.

6. Работа с NATS / JetStream KV:
KV использовать только для кэша, сессий и временных данных. Не использовать как основное хранилище бизнес-данных. Все операции Get/Put/Watch выполнять с таймаутами, чтобы не блокировать gateway при деградации кластера.

7. CI/CD и деплой:
При rolling update через Nomad учитывать, что во время деплоя часть инстансов может быть временно недоступна. Gateway должен корректно обрабатывать retry с ограничением, без каскадного усиления нагрузки.

8. Мониторинг:
Основной контроль состояния системы — через метрики NATS, задержки JetStream и ошибки timeout в gateway. Рост таймаутов = ранний сигнал деградации транспортного слоя.

9. Ресурсные ограничения:
При приближении к лимитам RAM возможны нестабильности NATS и JetStream из-за давления на память и disk I/O. Лимиты не защищают от кратковременных пиков.

### Соглашения

**`cmd/`**

Каждая папка соответствует одному бинарнику. Имя папки = имя исполняемого файла. `main.go` внутри должен быть минимальным: инициализация конфига, подключение к NATS/БД, запуск сервиса, ожидание сигнала завершения. Никакой бизнес-логики в `main.go`.

Все сервисы собираются одной командой: `go build ./cmd/...`

**`internal/`**

Код, который не должен импортироваться снаружи модуля. Go запрещает это на уровне компилятора.

- `internal/platform/` — инфраструктурные обёртки (NATS, БД и т.п.), общие для нескольких сервисов.
- `internal/services/{name}/` — логика конкретного сервиса: конфиг, обработчики NATS-сообщений, вспомогательные типы.
- `internal/middleware/` — переиспользуемые NATS-middleware (аутентификация, логирование и т.п.).

**`utils/`**

Лёгкие хелперы без бизнес-логики и без внешних зависимостей. Если утилита требует импорта `internal/` — она не принадлежит `utils/`.

**`deployments/`**

Конфигурационные файлы для инфраструктуры. Не содержит Go-кода.

- `services/` — шаблоны и описания сервисов, не зависящие от окружения (NATS-конфиг, Nomad job-файлы с переменными).
- `mode/dev/` — всё для локальной разработки: Docker Compose, dev-переменные, инструкции.
- `mode/prod/` — шаблон prod-переменных и инструкции по production-деплою.

**Именование демо-сервисов**

Все демонстрационные сервисы-примеры именуются с префиксом `x` (`xhttp`, `xauth`, `xws`). Это соглашение визуально отделяет шаблонный демо-код от production-сервисов конкретного проекта.
